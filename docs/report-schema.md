# Report Schema

This document describes the stable fields used by Model Integrity Audit JSON reports. Reports are designed for local review, CI evidence, and downstream tools that need a small explainable contract.

The schema is descriptive, not a formal JSON Schema file. New fields may be added in minor releases. Existing field meanings should not change without a README note.

## Privacy Defaults

- Endpoints are redacted by default as `<redacted-endpoint>`.
- API keys, bearer tokens, raw credentials, and private trace identifiers must not appear in reports.
- Generated reports are written under `reports/`, which is ignored by Git.
- Example reports live under `examples/reports/` and must use sanitized values only.
- `--show-endpoint` may include a sanitized endpoint origin for local-only troubleshooting.

## Main Audit Report

Generated by `check-api-quality-and-model-integrity.sh` in `quick` or `full` mode.

Common top-level fields:

- `timestamp`: ISO-8601 UTC report timestamp.
- `report_type`: Report kind, usually `api_quality_model_integrity`.
- `mode`: `quick` or `full`.
- `endpoint`: Redacted endpoint marker or sanitized endpoint origin.
- `sanitized`: Boolean indicating whether privacy redaction rules were applied.
- `verdict`: Final behavioral verdict.
- `score`: Numeric confidence score when the mode supports scoring.
- `evidence`: Array of explainability records.
- `warnings`: Array of warning strings derived from evidence.
- `failed_controls`: Array of failed or review-needed control names.
- `recommendations`: Array of recommended next actions.

Quick mode may also include:

- `target_seconds`: Intended quick-check runtime budget.
- `models`: Requested model list.
- `results`: Per-model request observations.

Full mode may also include:

- `samples`: Sample count per model.
- `baseline`: Baseline model used for comparison.
- `results`: Per-model aggregate observations.
- `comparisons`: Similarity or fingerprint comparisons.

## Focused GPT-5.5 Probe Report

Generated by `scripts/probe-gpt55-authenticity.sh`.

Common top-level fields:

- `timestamp`: ISO-8601 UTC report timestamp.
- `report_type`: Report kind, usually `gpt55_authenticity_probe`.
- `model`: Requested model id.
- `relay_base_url`: Redacted or sanitized relay base URL.
- `responses_url`: Redacted or sanitized Responses API URL.
- `official_base_url`: Redacted or sanitized official base URL when present.
- `sanitized`: Boolean indicating whether privacy redaction rules were applied.
- `score`: Numeric behavioral score.
- `verdict`: Final behavioral verdict.
- `checks`: Structured control results.
- `evidence`: Array of explainability records.
- `warnings`: Array of warning strings derived from evidence.
- `failed_controls`: Array of failed or review-needed control names.
- `recommendations`: Array of recommended next actions.

## Evidence Records

Each evidence record explains one check in human-readable form.

Fields:

- `level`: `info`, `warning`, or `error`.
- `control`: Stable control name, such as `valid_http`, `model_echo`, `negative_control`, or `usage_visibility`.
- `message`: Short explanation for humans.
- `value`: Optional supporting value. This must not contain secrets.

Consumers should treat unknown `control` names as informational unless `level` is `error`.

## Verdicts

Known verdict values include:

- `likely_real_gpt55_route`: The route passed the implemented behavioral checks.
- `suspicious_or_unstable`: The route failed important checks or looked unstable.
- `inconclusive`: The run did not produce enough evidence for a stronger call.
- `pass`: Generic success for non-model-specific reports.
- `warning`: Completed with review-needed signals.
- `fail`: Failed core controls.

These verdicts are behavioral evidence, not cryptographic proof of backend identity.

## Consumer Guidance

- Use `verdict`, `score`, `warnings`, and `failed_controls` together. Do not rely on score alone.
- Treat missing `usage` or model echo mismatch as review signals, not automatic proof of spoofing.
- Treat timeout and malformed JSON failures as endpoint reliability evidence.
- For billing, procurement, or incident response, combine reports with provider logs, billing exports, and independent operational checks.
